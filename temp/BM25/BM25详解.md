# BM25 算法详解：搜索引擎如何判断"哪个网页最相关"

> 本文面向计算机小白，用生活化的比喻讲清楚 BM25 是什么、怎么工作、有什么用。

---

## 一、一句话回答：BM25 是什么？

**BM25（Best Matching 25）是一种"打分公式"，搜索引擎用它来给文档打分，分数越高，说明这个文档和你的搜索词越相关。**

你在 Google、淘宝、Elasticsearch 里搜索时，背后大概率就有一个 BM25（或它的变体）在默默工作，帮你把"最相关的结果"排在前面。

---

## 二、用一个生活比喻理解它

想象你是一个图书管理员，有人来问你："我想找一本讲**猫的饮食**的书。"

你面前有 100 本书，你怎么判断哪本最相关？你可能会这样想：

| 你会考虑的问题 | 对应 BM25 的概念 |
|---|---|
| 这本书里"猫"和"饮食"这两个词出现了多少次？ | **词频 TF**（出现越多，可能越相关） |
| "猫"这个词在所有书里常见吗？"饮食"呢？ | **逆文档频率 IDF**（越罕见的词，越有区分度） |
| 这本书有多厚？是不是一本超级厚的百科全书？ | **文档长度归一化**（太厚的书，词频高也不稀奇） |

BM25 就是把这三个因素，用一个数学公式综合起来，给每本书打一个分数。

---

## 三、BM25 要解决什么问题？

在 BM25 之前，人们用 **TF-IDF** 来打分。但 TF-IDF 有两个明显的缺点：

### 缺点 1：词频"越多越好"没有上限

TF-IDF 认为，一个词在文档里出现 100 次，比出现 10 次"好 10 倍"。

但现实中，一篇讲猫的文章里出现 50 次"猫"和出现 100 次"猫"，相关性其实差不多——**"猫"这个词的边际效益是递减的**。

> 就像吃包子：第一个最香，第五个还行，第十个就撑了。词频也是这样，**越多不等于越好，会"饱和"**。

### 缺点 2：长文档天然占便宜

一篇 10000 字的长文，任何词出现的次数都比一篇 500 字的短文多。但这不代表长文更相关——可能只是因为它"话多"。

BM25 引入了**文档长度归一化**，让长文档不会因为"话多"就占便宜。

---

## 四、BM25 的核心公式（别怕，我们一步步拆解）

BM25 给一个文档 $D$ 和查询 $Q$ 的打分公式是：

$$
\text{score}(D, Q) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\text{avgdl}}\right)}
$$

看起来很吓人？别急，我们把它拆成三块：

### 第一块：IDF —— 这个词有多"稀有"

$$
\text{IDF}(q_i) = \ln\left(\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5} + 1\right)
$$

- $N$：总共有多少篇文档
- $n(q_i)$：有多少篇文档包含这个词

**通俗解释**：如果 100 本书里 99 本都提到"的"这个字，那"的"几乎没有区分度，IDF 接近 0。但如果只有 2 本书提到"量子力学"，那"量子力学"的 IDF 就很高——**越稀有的词，越能帮你精准定位**。

### 第二块：TF 的"饱和"处理 —— 词频不是越多越好

$$
\frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (\cdots)}
$$

- $f(q_i, D)$：这个词在这篇文档里出现了几次

**通俗解释**：这个分式的巧妙之处在于——随着词频 $f$ 增大，分数会趋近于一个上限 $(k_1 + 1)$，**永远不会超过它**。

> 就像往杯子里倒水，杯子容量是固定的（$k_1 + 1$），倒得越多越满，但永远不会溢出。

**$k_1$ 参数**控制"饱和速度"：
- $k_1$ 大（比如 2.0）：词频影响更持久，出现很多次还在涨分
- $k_1$ 小（比如 0.5）：很快饱和，出现两三次就到顶了
- 通常取值 **1.2 ~ 2.0**

### 第三块：文档长度归一化 —— 别让"话多"的文档占便宜

$$
\left(1 - b + b \cdot \frac{|D|}{\text{avgdl}}\right)
$$

- $|D|$：这篇文档有多长（多少个词）
- $\text{avgdl}$：所有文档的平均长度

**通俗解释**：
- 如果文档比平均长度长（$|D| > \text{avgdl}$），这个因子变大，**拉低分数**（惩罚长文档）
- 如果文档比平均长度短，这个因子变小，**相对提升分数**

**$b$ 参数**控制"惩罚力度"：
- $b = 1$：完全按长度比例归一化（严格惩罚长文档）
- $b = 0$：完全不考虑长度（不惩罚）
- 通常取值 **0.75**

## 五、一个完整的例子

假设你有 3 篇文档，用户搜索 **"猫 饮食"**：

| 文档 | 内容 | 长度 |
|---|---|---|
| 文档 A | "猫的饮食很重要。猫需要吃肉。" | 10 词 |
| 文档 B | "猫猫猫猫猫猫猫猫猫猫饮食饮食" | 12 词（堆砌关键词） |
| 文档 C | "一本关于动物学的百科全书，其中有一章提到猫的饮食习惯……"（很长） | 500 词 |

直觉上，**文档 A 最相关**——它简洁、直接地讲了猫的饮食。

BM25 会怎么打分？

- **文档 B**：虽然"猫"出现很多次，但由于 TF 饱和效应，多出来的"猫"不会加多少分。而且关键词堆砌很明显，分数不会太高。
- **文档 C**：虽然也提到了"猫"和"饮食"，但因为文档太长（500 词），长度归一化会拉低它的分数。
- **文档 A**：词频适中、长度正常，最终得分最高。

这就是 BM25 的威力——**它像一个人工智能裁判，综合考虑"词频、词的稀有度、文档长度"，给出一个合理的 relevance 分数**。

## 六、BM25 和 TF-IDF 的对比

| 特性 | TF-IDF | BM25 |
|---|---|---|
| 词频处理 | 线性增长（越多越好，无上限） | 饱和增长（有上限，边际递减） |
| 文档长度 | 不考虑 | 归一化处理（惩罚过长的文档） |
| 参数可调 | 无 | 有 $k_1$ 和 $b$ 两个参数 |
| 效果 | 基础 | 更好（业界标准） |
| 复杂度 | 简单 | 略复杂，但计算量差不多 |

一句话总结：**BM25 = TF-IDF 的"升级版"，加了"饱和"和"长度归一化"两个改进**。

---

## 七、BM25 有什么用？

### 1. 搜索引擎（最核心的应用）

- **Elasticsearch / Lucene / Solr**：这些主流搜索引擎的默认打分算法就是 BM25（或其变体）
- 你在电商网站搜索商品、在文档站搜索文章、在 App 里搜索内容，背后很可能就是 BM25

### 2. 全文检索数据库

- PostgreSQL 的全文搜索支持 BM25 排序
- MongoDB Atlas Search 也使用 BM25

### 3. RAG（检索增强生成）中的检索阶段

- 在大模型应用中，RAG 的第一步是"从知识库里找回最相关的文档"
- 这个"找回"步骤，很多系统用的就是 BM25（或 BM25 + 向量检索的混合方案）

### 4. 问答系统

- 给定一个问题，从 FAQ 库里用 BM25 找出最相关的候选问答

## 八、用 Python 写一个最简单的 BM25

不用任何库，30 行代码就能实现一个基础版 BM25：

```python
import math
from collections import Counter

class SimpleBM25:
    def __init__(self, documents, k1=1.5, b=0.75):
        """
        documents: 文档列表，每个文档是一个词列表
        例如: [["猫", "饮食", "重要"], ["狗", "跑步"], ...]
        """
        self.documents = documents
        self.k1 = k1
        self.b = b
        self.doc_len = [len(doc) for doc in documents]  # 每篇文档长度
        self.avgdl = sum(self.doc_len) / len(documents)  # 平均文档长度
        self.N = len(documents)  # 文档总数

        # 统计每个词出现在多少篇文档中
        self.doc_freq = Counter()
        for doc in documents:
            for word in set(doc):  # set 去重，每篇文档每个词只算一次
                self.doc_freq[word] += 1

    def idf(self, word):
        """计算一个词的 IDF 值"""
        n = self.doc_freq.get(word, 0)
        return math.log((self.N - n + 0.5) / (n + 0.5) + 1)

    def score(self, query, doc_idx):
        """计算 query 对第 doc_idx 篇文档的 BM25 分数"""
        doc = self.documents[doc_idx]
        doc_counter = Counter(doc)
        score = 0.0

        for word in query:
            if word not in doc_counter:
                continue
            f = doc_counter[word]  # 词频
            idf = self.idf(word)
            # TF 饱和 + 文档长度归一化
            tf_component = (f * (self.k1 + 1)) / (
                f + self.k1 * (1 - self.b + self.b * self.doc_len[doc_idx] / self.avgdl)
            )
            score += idf * tf_component

        return score

    def search(self, query, top_k=3):
        """搜索，返回最相关的 top_k 篇文档"""
        scores = [(i, self.score(query, i)) for i in range(self.N)]
        scores.sort(key=lambda x: x[1], reverse=True)
        return scores[:top_k]


# ===== 试一试 =====
docs = [
    ["猫", "饮食", "很重要", "猫", "需要", "吃", "肉"],
    ["狗", "喜欢", "跑步", "狗", "是", "忠实", "的", "朋友"],
    ["猫", "和", "狗", "都", "是", "常见", "的", "宠物", "饮食", "各", "不同"],
    ["量子力学", "是", "物理学", "的", "一个", "分支"],
]

bm25 = SimpleBM25(docs)
query = ["猫", "饮食"]
results = bm25.search(query, top_k=3)

print("搜索词:", query)
for doc_idx, score in results:
    print(f"  文档 {doc_idx} (分数: {score:.4f}): {docs[doc_idx]}")
```

运行后你会看到文档 0（"猫的饮食很重要"）得分最高，文档 2（"猫和狗都是宠物"）次之，文档 3（量子力学）完全不相关。

## 九、BM25 的局限性

BM25 很强，但不是万能的：

| 局限 | 说明 | 解决方案 |
|---|---|---|
| **只看关键词匹配** | 不理解语义，"开心"和"快乐"对它来说是两个不同的词 | 用向量检索（Embedding）补充语义理解 |
| **不理解词序** | "猫吃鱼"和"鱼吃猫"对 BM25 来说一样 | 用深度学习模型（如 BERT）重排序 |
| **不处理同义词** | 搜"手机"找不到只写"智能手机"的文档 | 用同义词表扩展查询，或用向量检索 |
| **需要调参** | $k_1$ 和 $b$ 需要根据数据调整 | 用验证集调参，或用默认值（1.2, 0.75） |

现代搜索系统通常采用**混合检索**：BM25（关键词匹配）+ 向量检索（语义匹配），取两者之长。

## 十、总结

| 问题 | 回答 |
|---|---|
| BM25 是什么？ | 一个给文档打分的公式，判断文档和搜索词有多相关 |
| 核心思想？ | 综合考虑三个因素：词频（会饱和）、词的稀有度（IDF）、文档长度（归一化） |
| 和 TF-IDF 的区别？ | BM25 加了"词频饱和"和"长度归一化"两个改进 |
| 两个关键参数？ | $k_1$（控制词频饱和速度，常用 1.2~2.0）、$b$（控制长度惩罚力度，常用 0.75） |
| 用在哪？ | Elasticsearch、Solr、Lucene、PostgreSQL、RAG 检索阶段等 |
| 局限性？ | 不理解语义、同义词、词序，需配合向量检索使用 |

> **一句话记忆**：BM25 就像一个聪明的图书管理员——它知道"越稀有的词越重要"（IDF），知道"说太多遍就不值钱了"（TF 饱和），还知道"话多的人不能信"（长度归一化）。
