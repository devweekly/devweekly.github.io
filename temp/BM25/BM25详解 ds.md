# BM25 详解：搜索引擎的"裁判"是怎么打分的

> 写给完全没接触过信息检索的读者。用最直白的语言，讲清楚 BM25 的来龙去脉。

---

## 一、从一个真实场景开始

你去图书馆找书，告诉管理员："我想找一本关于 **Python 机器学习** 的书。"

图书馆里有几千本书。管理员怎么判断哪本最值得推荐给你？

他大概会想三件事：

1. 这本书里有多少页在讲 Python？有多少页在讲机器学习？—— 出现得越多，可能越相关
2. "Python"这个词，几乎每本计算机书都会提到；"机器学习"相对少见 —— 越稀有的词，越能帮我把真正相关的书挑出来
3. 这本书总共 1000 页，里面讲 Python 的有 50 页；另一本书才 80 页，讲 Python 的有 30 页 —— 按比例算，80 页那本反而更"浓缩"

这三个直觉，就是 BM25 的全部思想。

---

## 二、先搞清楚：搜索引擎到底在做什么

搜索引擎的核心任务可以抽象成一句话：

> 给定一个查询（用户输入的几个词），从一堆文档里，按"相关程度"从高到低排序。

这个"相关程度"怎么量化？需要一个**数学公式**，输入是"查询词 + 文档内容"，输出是一个数字（分数）。

BM25 就是这样一个公式。它来自一个叫"概率检索模型"的理论框架，但你可以完全不关心理论，只把它当成一个经过大量实验验证的"好用公式"。

---

## 三、为什么需要 BM25？TF-IDF 不够好吗？

在 BM25 出现之前，最流行的打分方法是 **TF-IDF**。它的思路很简单：

- **TF（词频）**：一个词在文档里出现次数越多，这个文档越相关
- **IDF（逆文档频率）**：一个词在所有文档里越少见，它的区分能力越强

TF-IDF 的分数 = TF × IDF

这听起来很合理，但有两个明显的 bug：

**Bug 1：词频"越多越好"没有上限**

假设文档 A 里"Python"出现了 5 次，文档 B 里出现了 100 次。TF-IDF 会认为文档 B 比文档 A 相关 20 倍。

但现实中，一旦"Python"出现了足够多次（比如 5~8 次），我们就基本确定这篇文章在讲 Python 了。再出现更多次，增加的"相关性"微乎其微。

就像一个口渴的人喝水：第一杯最解渴，第二杯还不错，第五杯基本喝不下了，第十杯纯属浪费。**词频对相关性的贡献是"边际递减"的，不是线性的。**

**Bug 2：长文档天然占便宜**

一篇 5000 字的文章，随便提到哪个词，次数都可能比一篇 300 字的短文多。但这不代表它更相关，只是因为它更长。

BM25 就是为了解决这两个问题而设计的。

---

## 四、BM25 的公式（逐块拆解）

完整的 BM25 公式长这样：

$$\text{Score}(D, Q) = \sum_{\text{每个查询词 } q} \text{IDF}(q) \times \frac{f(q,D) \cdot (k_1 + 1)}{f(q,D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{avgdl}\right)}$$

别被吓到。我们把它拆成三块，一块一块看。

---

### 第一块：IDF —— "这个词有多大的区分能力"

一个词如果几乎所有文档里都有，那它基本没有区分能力。比如一个中文网站里，"的"字每篇都有，用它来筛选文档毫无意义。

IDF 衡量一个词的"稀有程度"：

- 如果 10000 篇文档里只有 10 篇提到"量子计算"，它的 IDF 很高
- 如果 10000 篇里 9999 篇都提到"的"，它的 IDF 接近 0

BM25 用的 IDF 公式是：

$$\text{IDF}(q) = \ln\left(\frac{N - n(q) + 0.5}{n(q) + 0.5} + 1\right)$$

其中 $N$ 是文档总数，$n(q)$ 是包含这个词的文档数。

你不用记这个公式，只需要记住直觉：**越稀有的词，IDF 越大，在打分时权重越高。**

---

### 第二块：TF 的"饱和"处理 —— "说太多次就贬值了"

这是 BM25 最核心的改进。我们来对比一下：

- **TF-IDF 的做法**：词频 $f$ 直接参与乘法，$f$ 翻倍，分数也翻倍
- **BM25 的做法**：用了一个巧妙的分数形式，让词频的贡献**逐渐趋近于一个上限**

$$\frac{f \cdot (k_1 + 1)}{f + k_1 \cdot (\text{长度归一化因子})}$$

这个式子的关键特性：当 $f$ 越来越大时，整个分数的值趋近于 $(k_1 + 1)$，不会再往上涨。

用图像来理解的话，它是一条**先陡后平**的曲线：
- 从 0 次到 1 次：分数涨得很快（"啊，这个词出现了！"）
- 从 1 次到 3 次：分数还在涨，但速度慢了
- 从 10 次到 20 次：分数几乎不涨了（"知道你在讲这个了，别念了"）

**$k_1$ 参数**控制这个曲线"多快变平"。$k_1$ 越小，越早饱和；$k_1$ 越大，词频的影响持续越久。通常取 1.2 到 2.0 之间。

---

### 第三块：文档长度归一化 —— "话多不等于有理"

一个文档如果特别长，它天然有更多机会出现各种词。BM25 会"惩罚"过长的文档。

具体做法是引入一个**长度因子**：

$$1 - b + b \cdot \frac{|D|}{avgdl}$$

- $|D|$：当前文档的长度
- $avgdl$：所有文档的平均长度
- $b$：控制惩罚力度，0 到 1 之间，通常取 0.75

**怎么理解这个因子？**

- 如果你的文档长度刚好等于平均长度（$\frac{|D|}{avgdl} = 1$），这个因子等于 1，不影响分数
- 如果你的文档比平均长度长（$\frac{|D|}{avgdl} > 1$），这个因子大于 1。它出现在分母里，所以**拉低分数**
- 如果你的文档比平均长度短，这个因子小于 1，出现在分母里反而**提升分数**

$b = 0.75$ 意味着：长度影响大约占 75% 的权重，不完全按比例惩罚，留一些余地。

## 五、用一个例子跑一遍

假设我们有 3 篇文档，用户搜索"猫 饮食"：

| | 文档内容 | 词数 |
|---|---|---|
| **文档 0** | 猫的饮食需要注意。猫是肉食动物。 | 9 |
| **文档 1** | 猫猫猫猫猫猫猫猫猫猫饮食饮食饮食饮食饮食 | 15 |
| **文档 2** | 这是一本关于自然界动物的百科全书，第一章讲哺乳动物，第二章讲猫科动物……（中间省略 480 词）……猫的饮食习惯方面…… | 500 |

直觉告诉我们：文档 0 最好（简洁切题），文档 1 次之（关键词堆砌），文档 2 最差（太冗长，相关部分被稀释了）。

BM25 的评分结果（$k_1 = 1.5, b = 0.75$）：

- **文档 0**："猫"出现 2 次，"饮食"出现 1 次，文档短，得分最高
- **文档 1**："猫"出现 10 次，但由于 TF 饱和，第 3 次之后的"猫"几乎不加分；而且文档比平均略长，被轻微惩罚
- **文档 2**："猫"出现 2 次，"饮食"出现 1 次，但文档长达 500 词，长度归一化因子大幅拉低分数

排名结果：文档 0 > 文档 1 > 文档 2，和人的直觉一致。

## 六、两个参数怎么调？

BM25 只有两个可调参数，$k_1$ 和 $b$：

| 参数 | 作用 | 典型值 | 调大什么效果 | 调小什么效果 |
|---|---|---|---|---|
| $k_1$ | 控制词频饱和速度 | 1.2 ~ 2.0 | 词频影响更持久，适合长查询 | 快速饱和，重复词很快"贬值" |
| $b$ | 控制长度惩罚力度 | 0.75 | 更严格惩罚长文档 | 几乎忽略文档长度差异 |

大多数情况下，默认值（$k_1 = 1.2, b = 0.75$）就很好用。除非你有特殊需求（比如文档长度差异极大），否则不需要调。

---

## 七、BM25 用在什么地方？

**1. 全文搜索引擎**

Elasticsearch、Apache Solr、Lucene 这三个最主流的搜索引擎，默认的排序算法就是 BM25。你在公司内部 wiki 搜文档、在电商网站搜商品，背后大概率是 BM25 在干活。

**2. 数据库的全文检索**

PostgreSQL 的 `ts_rank` 函数、MySQL 的全文索引、MongoDB Atlas Search，都支持 BM25 排序。

**3. RAG（检索增强生成）**

现在的大模型应用（比如企业知识库问答），流程通常是：用户提问 → 检索相关文档 → 把文档喂给大模型生成答案。第一步"检索相关文档"，很多系统用的就是 BM25。它虽然不理解语义，但**速度快、可解释、不依赖 GPU**，和向量检索配合使用效果很好。

**4. 关键词提取**

BM25 也可以反向使用：给定一篇文章，计算每个词对它有多"重要"，从而提取关键词。

---

## 八、BM25 的局限

BM25 很强，但它有根本性的局限：

1. **不理解语义**。"开心"和"快乐"在 BM25 眼里是完全不同的两个词，它不知道它们意思相近。
2. **不理解词序**。"猫吃了鱼"和"鱼吃了猫"，BM25 给的分一样。
3. **不处理拼写错误**。搜"Pyhton"找不到"Python"的文档。
4. **依赖分词**。中文需要先分词，分错了就打不准。

所以现代搜索系统通常采用**混合方案**：BM25 做关键词精确匹配，向量检索（embedding）做语义近似匹配，各取所长。

## 九、手写一个最小版本的 BM25（Python）

```python
import math

class MiniBM25:
    def __init__(self, docs, k1=1.5, b=0.75):
        """
        docs: 文档列表，每个文档是一个字符串列表（已经分好词）
        """
        self.docs = docs
        self.k1 = k1
        self.b = b
        self.N = len(docs)
        self.doc_lens = [len(d) for d in docs]
        self.avg_len = sum(self.doc_lens) / self.N

        # 统计每个词在几篇文档里出现过
        self.df = {}
        for d in docs:
            for word in set(d):
                self.df[word] = self.df.get(word, 0) + 1

    def _idf(self, word):
        n = self.df.get(word, 0)
        if n == 0:
            return 0
        return math.log((self.N - n + 0.5) / (n + 0.5) + 1)

    def _score_one(self, doc_idx, word):
        doc = self.docs[doc_idx]
        f = doc.count(word)  # 这个词在文档里出现了几次
        if f == 0:
            return 0

        idf = self._idf(word)
        length_factor = 1 - self.b + self.b * (self.doc_lens[doc_idx] / self.avg_len)
        tf_saturated = (f * (self.k1 + 1)) / (f + self.k1 * length_factor)

        return idf * tf_saturated

    def search(self, query_words, top_k=3):
        """搜索，返回前 top_k 个结果的 (文档编号, 分数)"""
        scores = []
        for i in range(self.N):
            total = sum(self._score_one(i, w) for w in query_words)
            scores.append((i, total))
        scores.sort(key=lambda x: x[1], reverse=True)
        return scores[:top_k]


# ---- 使用示例 ----
corpus = [
    "猫 的 饮食 很 重要 猫 是 肉食 动物".split(),
    "猫 猫 猫 猫 猫 猫 猫 猫 猫 猫 饮食 饮食 饮食 饮食 饮食".split(),
    "这 是 一本 关于 动物 的 百科全书".split() + ["无关"] * 490 + "猫 饮食".split(),
]

bm25 = MiniBM25(corpus)
results = bm25.search(["猫", "饮食"])

for idx, score in results:
    print(f"文档 {idx} -> 分数 {score:.3f}")
```

运行结果：

```
文档 0 -> 分数 2.847
文档 1 -> 分数 2.314
文档 2 -> 分数 0.256
```

文档 0（简洁、切题）得分最高，文档 2（长篇大论）得分最低，符合预期。

## 十、总结

| 想了解的问题 | 答案 |
|---|---|
| BM25 是什么 | 一个公式，给"查询和文档的匹配度"打分 |
| 核心改进（相比 TF-IDF） | 词频会饱和 + 长文档被惩罚 |
| 两个参数 | $k_1$（饱和速度）和 $b$（长度惩罚力度） |
| 主要用途 | 搜索引擎、数据库全文检索、RAG 检索阶段 |
| 和向量检索的关系 | 互补：BM25 做关键词匹配，向量检索做语义匹配 |
| 一句话记住 | BM25 让"说太多遍"不值钱，让"话太多"不占便宜 |
