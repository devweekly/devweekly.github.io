# BM25 详解：搜索引擎怎么知道哪篇文章更配你的问题

> 写给零基础的读者。不背公式，先建立直觉，再看实现。

---

## 一、从一个快递站的场景开始

想象你开了一家快递站，每天有上万个包裹进来。客户来找快递时只说一句话："我的包裹里有**红色围巾**和**手套**。"

你要从上万个包裹里找出最可能是他的那几个。你会怎么判断？

你肯定会想三件事：

1. **匹配程度**：包裹里"红色围巾"和"手套"各出现了几次？出现次数多的更可能是对的。
2. **词的分辨力**：如果仓库里一半的包裹都写了"红色围巾"，那这个词就没那么值钱；如果只有 20 个包裹写了"手套"，那"手套"这个词就很值钱。
3. **包裹大小**：一个巨大的集装箱，里面零散地放了条红色围巾；另一个小纸盒，打开就是红色围巾加手套。显然小纸盒更"浓缩"、更相关。

BM25 干的就是这件事：它给每个包裹算一个分数，分数越高，越可能是客户要找的。

搜索引擎里，"包裹"就是网页、文章、文档；"客户说的话"就是搜索框里的关键词。BM25 就是那个帮你快速分拣的"智能评分员"。

---

## 二、BM25 为什么会出现？

在它之前，搜索引擎用得最多的是 **TF-IDF**。它把两件事乘在一起：

- **TF（Term Frequency）**：关键词在文档里出现多少次
- **IDF（Inverse Document Frequency）**：关键词在所有文档里有多稀有

这个想法很朴素，但有两个硬伤。

### 硬伤一：关键词出现越多，分数越高，没有天花板

假设你搜"围巾"：

- 文档 A："我今天买了一条红色围巾。" —— "围巾"出现 1 次
- 文档 B："围巾围巾围巾围巾围巾围巾围巾围巾围巾……" —— "围巾"出现 100 次

按 TF-IDF，文档 B 的分数可能是文档 A 的 100 倍。但你心里清楚，文档 B 大概率是垃圾堆砌，不是真正相关的内容。

TF-IDF 就像一个不会腻的评委，你重复念一个词一百遍，他就觉得你说了一百倍重要的话。这显然不对。

### 硬伤二：大包裹天然占便宜

一个 5000 字的百科全书词条，随便什么词都能出现好几次；而一个 100 字的简短回答，即使句句切题，总词频也拼不过前者。

这不公平。我们不能因为某个文档"话多"，就认为它更相关。

BM25 就是来解决这两个问题的：

- 让关键词的加分**有天花板**（饱和）
- 让文档长度**参与校正**（太长要扣一点分）

---

## 三、BM25 的三个核心直觉

记住这三句话，你就懂了 BM25 的灵魂：

### 1. 稀有的词更值钱

仓库里只有 3 个包裹写了"羊绒"，这个词一旦出现，几乎可以确定是个强信号。仓库里有 3000 个包裹写了"红色"，那这个词就普普通通。

BM25 用 **IDF** 来表达这个思想。

### 2. 重复念太多遍，边际价值会下降

第一次出现"围巾"，分数猛涨："哦，这个包裹确实在说围巾。"
第二次出现，分数还在涨，但涨得少了。
第十次出现，分数基本不动了："知道了知道了，你别念叨了。"

BM25 用 **TF 饱和函数** 来表达这个思想。

### 3. 小包裹更浓缩，大包裹要被"打折"

同样的关键词，出现在 100 字的小文档里，和在 5000 字的大文档里，意义完全不同。小文档更可能整篇都在讲这件事；大文档可能只是顺带提到。

BM25 用 **文档长度归一化** 来表达这个思想。

## 四、公式拆解（不用背，理解结构就行）

BM25 给一个查询 $Q$ 和文档 $D$ 打的总分是：

$$\text{Score}(D, Q) = \sum_{q \in Q} \text{IDF}(q) \times \text{TF-Score}(q, D)$$

查询里有几个关键词，就把每个词的得分加起来。比如搜"红色围巾 手套"，就把"红色围巾"的得分和"手套"的得分相加。

### IDF 部分

$$\text{IDF}(q) = \ln\left(\frac{N - n(q) + 0.5}{n(q) + 0.5} + 1\right)$$

- $N$：仓库里包裹总数
- $n(q)$：写了这个词的包裹数量

结果：越稀有的词，IDF 越大。"手套"如果很少见，IDF 就高；"红色"如果随处可见，IDF 就低。

### TF-Score 部分

$$\text{TF-Score}(q, D) = \frac{f(q, D) \times (k_1 + 1)}{f(q, D) + k_1 \times \left(1 - b + b \times \frac{|D|}{avgdl}\right)}$$

- $f(q, D)$：关键词在文档里出现几次
- $|D|$：这个文档有多长
- $avgdl$：所有文档的平均长度
- $k_1$：控制"重复多少次开始饱和"，通常 1.2 ~ 2.0
- $b$：控制"长度惩罚有多狠"，通常 0.75

**这个分数有两个秘密：**

1. 当 $f(q, D)$ 越来越大时，分子分母同步增长，整个分数会趋近于一个最大值。也就是说，重复再多遍，也不会无限加分。
2. 文档越长，分母里的长度因子越大，分数就会被压下来。

## 五、一个具体例子：给三篇文章打分

假设仓库里有三篇文章，用户搜："**咖啡 烘焙**"

| 编号 | 文章 | 长度 |
|------|------|------|
| A | "咖啡的烘焙程度决定了风味。浅烘焙的咖啡酸度高，深烘焙的咖啡苦味重。" | 28 字 |
| B | "咖啡咖啡咖啡咖啡咖啡咖啡咖啡咖啡咖啡咖啡烘焙烘焙烘焙烘焙烘焙烘焙" | 30 字 |
| C | "这是一本关于饮食文化的书，第一章讲茶，第二章讲酒，第三章讲谷物……第十章末尾提到咖啡的烘焙历史。" | 500 字 |

### 人的直觉

- **A 最好**：整段都在讲咖啡烘焙，直接切题。
- **B 不好**：关键词堆砌，像垃圾内容。
- **C 一般**：虽然提到了，但淹没在长篇大论里，相关性被稀释了。

### BM25 会怎么做

- **"咖啡"和"烘焙"的 IDF**：假设这两个词都不算特别常见，IDF 给一个中等偏高的分数。
- **文档 A**：两个词各出现 2 次，文档长度接近平均。分数最高。
- **文档 B**："咖啡"出现 10 次，但由于 TF 饱和，第 3 次以后基本不再加分；"烘焙"同理。堆砌行为被抑制。
- **文档 C**：两个词各出现 1 次，但文档长达 500 字，长度因子会把它压到很低。

结果：A > B > C。和人的判断一致。

---

## 六、参数 $k_1$ 和 $b$ 是干什么的？

### $k_1$：控制"重复多少次开始腻"

- $k_1 = 0.5$：很快饱和。出现 2~3 次就顶天了。适合希望关键词"点到为止"的场景。
- $k_1 = 2.0$：饱和慢。出现 10 次还能继续加分。适合长文档、长查询。

默认值通常是 1.2 到 1.5。

### $b$：控制"长文档要扣多少分"

- $b = 0$：完全不考虑长度。长短文档公平竞争。
- $b = 1$：严格按长度比例惩罚。长文档会被压得很厉害。
- $b = 0.75$：折中方案，也是默认值。

你可以把 $b$ 理解为"长度惩罚的力度旋钮"。

## 七、BM25 有什么用？

### 1. 搜索引擎

Elasticsearch、Solr、Lucene 都默认使用 BM25。你用的公司内部搜索、电商商品搜索、文档检索，基本都离不开它。

### 2. 数据库全文检索

PostgreSQL、MySQL、MongoDB Atlas Search 都提供基于 BM25 的全文检索排序。

### 3. RAG 系统

现在很火的"大模型 + 知识库"应用，第一步通常是用 BM25 从知识库里快速召回相关文档，再交给大模型去理解和生成答案。

BM25 速度快、不吃 GPU、结果可解释，是 RAG 检索阶段的经典选择。

### 4. 关键词推荐

也可以反过来用：给定一篇文章，计算每个词对这篇文章的重要性，选出最有代表性的关键词。

---

## 八、BM25 不是万能的

它有三个明显短板：

### 1. 不理解语义

"开心"和"快乐"在 BM25 眼里是两个完全不同的词。它不知道它们是同义词。要解决这个问题，需要向量检索或者同义词表。

### 2. 不理解词序和语法

"猫吃了鱼"和"鱼吃了猫"在 BM25 里得分一样。它把文档当成一个"词袋子"，只看有没有，不看顺序。

### 3. 对拼写错误没辙

搜"coffe"找不到"coffee"。需要额外的拼写纠正模块。

所以现代搜索系统通常是**混合方案**：BM25 负责精确的关键词匹配，向量模型负责语义理解，两者结果合并后返回。

## 九、用 Python 手写一个 BM25

下面是一个不依赖任何第三方库的最小实现：

```python
import math

class BM25:
    def __init__(self, documents, k1=1.5, b=0.75):
        """
        documents: 已经分好词的文档列表，每个文档是 list[str]
        """
        self.k1 = k1
        self.b = b
        self.documents = documents
        self.doc_count = len(documents)
        self.doc_lengths = [len(doc) for doc in documents]
        self.avg_length = sum(self.doc_lengths) / self.doc_count

        # 统计每个词在多少篇文档里出现过
        self.doc_freq = {}
        for doc in documents:
            for word in set(doc):
                self.doc_freq[word] = self.doc_freq.get(word, 0) + 1

    def idf(self, word):
        n = self.doc_freq.get(word, 0)
        if n == 0:
            return 0
        return math.log((self.doc_count - n + 0.5) / (n + 0.5) + 1)

    def score(self, query_words, doc_index):
        """计算查询词对指定文档的 BM25 分数"""
        doc = self.documents[doc_index]
        total = 0

        for word in query_words:
            freq = doc.count(word)
            if freq == 0:
                continue

            idf_score = self.idf(word)
            length_factor = 1 - self.b + self.b * (self.doc_lengths[doc_index] / self.avg_length)
            tf_score = (freq * (self.k1 + 1)) / (freq + self.k1 * length_factor)

            total += idf_score * tf_score

        return total

    def search(self, query_words, top_k=3):
        """返回得分最高的 top_k 篇文档"""
        results = []
        for i in range(self.doc_count):
            s = self.score(query_words, i)
            results.append((i, s))
        results.sort(key=lambda x: x[1], reverse=True)
        return results[:top_k]


# ===== 示例 =====
corpus = [
    "咖啡 的 烘焙 程度 决定 了 风味".split(),
    "咖啡 咖啡 咖啡 咖啡 咖啡 咖啡 咖啡 咖啡 咖啡 咖啡 烘焙 烘焙 烘焙 烘焙 烘焙".split(),
    ["这", "是", "一", "本", "很", "长", "的", "书"] + ["饮食"] * 480 + ["咖啡", "烘焙"],
]

bm25 = BM25(corpus)
rank = bm25.search(["咖啡", "烘焙"])

for idx, score in rank:
    print(f"文档 {idx}: 分数 = {score:.3f}")
```

输出大概是：

```
文档 0: 分数 = 2.512
文档 1: 分数 = 1.987
文档 2: 分数 = 0.203
```

文档 0 胜出，文档 1 虽然堆砌了关键词，但因为 TF 饱和无法反超，文档 2 因为太长被压到最低。

## 十、一句话总结

BM25 是一个**既看关键词出现次数、又看关键词稀有程度、还会惩罚长篇大论**的打分公式。

它让搜索引擎明白：

- 稀有的词更有分量
- 重复太多遍会贬值
- 小文档更浓缩，大文档要打折扣

这就是它为什么能成为 Elasticsearch、Solr、Lucene 等主流搜索引擎默认排序算法的原因。
