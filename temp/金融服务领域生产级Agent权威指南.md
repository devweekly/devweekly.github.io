### 金融服务领域生产级Agent：来自摩根大通、Chime 和桥水基金的经验教训

Agent正在出现在研究桌面、顾问工具和面向会员的副驾驶中。将其投入生产环境的团队，大部分时间都花在了运营层面的工作上：提升可观测性、更早地将合规引入开发流程，以及缩短迭代循环。

金融服务行业在过去一年中已经从"有趣的演示"迈向了"面向客户"的阶段。取得进展的团队正将大部分时间用于缩短迭代循环。他们需要可解释的追踪链路、可依赖的评估体系，以及始终在线的运行时控制。

其中，两个关注点始终高悬于一切之上：**治理**和**安全**。我们接触的平台团队尤其关注三件事——开源模型、Agent评估和护栏机制。

我们通过以下团队观察到了这一趋势的展开：

- **摩根大通私人银行**：构建了 Ask DAVID，一个面向投资团队的内部多Agent研究助手
- **桥水基金**：系统化宏观对冲基金，构建了 PAT（袖珍分析师工具），部署给数百名投资者使用的内部研究Agent
- **Chime**：构建了 Jade，一个始终在线的金融副驾驶，旨在帮助会员处理财务问题和决策

每个团队都在解决不同的问题，但它们共同描绘了金融服务领域生产级Agent系统的清晰图景。

本指南将深入探讨行业中涌现的用例、团队正在攻克的技术和运营挑战，以及企业如何在整个Agent开发生命周期中使用 LangSmith 来构建、测试、部署、监控和治理这些系统。

### 1. 金融服务中涌现的用例

在构建Agent的金融服务团队中，几种模式正在反复涌现。

**内部研究助手**是最常见的起点。投资研究团队坐拥数十年的结构化数据、日益增长的非结构化资料（会议记录、电话、视频、演示文稿）以及过去需要人工专家才能操作的专有分析工具。一个能够跨这三者回答问题的Agent，可以将研究团队从过去占据大量时间的繁琐查找中解放出来，并为那些从未获得过白手套级别服务的客户提供可规模化的洞察。

这一类别也是最具野心的工具所在，包括将分散数据源整合到单一聊天界面的投资组合经理助手，以及用于生成和压力测试交易策略的交易助手。

**顾问和客户关系经理副驾驶**旨在直接支持面向客户的员工。一位金融顾问在实时会议中无法暂停去查阅研究报告，他们需要在几秒钟内获得答案，要针对他们正在交谈的客户进行个性化处理，并且要提供经得起推敲的引用来源。客户关系经理Agent是针对不同角色但形态相同的工具，在需要的时刻呈现正确的账户上下文。这也是编排和个性化变得更加重要的地方，因为相同的底层信息通常需要根据用户和上下文以不同方式呈现。

**面向会员的副驾驶**是风险最高的层级。当Agent直接与消费者就财务决策进行交互时，错误的容忍度会大大降低。团队需要围绕合规、升级和运行时行为建立更强的防护措施，这些系统才能在生产环境中获得信任。

**运营和合规副驾驶**是后台增长最快的类别，它们共享一个共同的形态：一个客户请求或案件涉及多个团队和系统，而Agent的工作是将这些分散的工作整合到一个可审查的地方。最常见的示例包括：

- **KYC 和 AML 调查**：了解你的客户和反洗钱调查是我们在整个行业中最为一致看到的用例。这项工作体量大、证据密集，天然适合由Agent从多个系统收集信息并汇总成调查员可以辩护的证据链。
- **客户入职QA**：一个审查入职案例并自动执行人工审批步骤的QA Agent。
- **客户服务摘要**：当多个团队处理同一个客户请求时，摘要Agent会写出发生的情况，让客户经理无需阅读全部交互记录即可掌握上下文。

**通过 MCP 的数据访问**是最新的模式。金融数据提供商将其产品暴露为 MCP 服务器，客户可以将研究和分析拉入 Claude、ChatGPT 或他们自己的自定义Agent中。这也改变了Agent体验的所在位置——在某些情况下Agent直接嵌入提供商的产品内部，而在其他情况下客户将数据带入自己的Agent环境。

在这五种模式中，技术问题是相似的：如何跨多个数据源编排工作？如何知道答案是正确的？以及如何发布一个不会被监管机构要求下线的产品？

### 2. 三个拥有生产级Agent的团队

#### 摩根大通私人银行：Ask DAVID

摩根大通私人银行的投资研究团队管理着成千上万只产品的列表，每只产品背后都有多年的数据支撑。来自顾问和投资团队支持客户的咨询不断涌入。过去回答这些问题意味着一个小型研究团队需要手动翻阅数据库、材料和文件。Ask DAVID（数据、分析、可视化、洞察、决策的首字母缩写）是该团队将研究工作流自动化、同时保持业务期望的准确性和可靠性水平的努力成果。

Ask DAVID 的第一个版本是基于 LangGraph 构建的多Agent系统。一个监督Agent与用户对话，持有短期和长期记忆，并将任务委派给三个专门的子Agent。

架构随时间逐步演变。团队从基础的 ReAct Agent开始，然后引入检索工作流，接着是带有单一监督者的多Agent系统，最终迁移到当前基于子图的架构。然后，他们开始使用按意图路由的子图。

现在，团队已经构建了 Ask DAVID 2.0。在 2.0 架构中，团队使用 LangChain 的开源 deepagents 包将监督者本身改造为**Deep Agent**。

Deep Agent的核心仍然是最简单的东西：一个在循环中运行、调用工具的 LLM。使它"深"而非"浅"的是四个要素的组合：

- 一个详细的系统提示，告诉Agent如何行事
- 一个用于分解和跟踪多步骤工作的规划工具
- 生成子Agent的能力，深入处理任务的各个部分
- Agent及其子Agent可以读写、用于笔记和交接的文件系统

朴素的工具调用Agent在长时、多步骤任务上往往会失败。Deep Agent模式——支撑 Deep Research 和 Claude Code 等工具的底层模式——正是让Agent能够规划复杂任务并在更长时间跨度内执行该计划的关键。子Agent使其能够深入而不污染主上下文，文件系统则给它提供了记笔记和移交工作的地方。

对于 Ask DAVID 而言，作为 Deep Agent 的监督者由摩根大通专有的私人银行**技能**驱动，这是其研究工作实际上如何完成的业务编码知识。这些技能让 Deep Agent 能够可靠地而非临时性地驾驭一支专门的Agent舰队，这正是可靠的规模化工作流和每个用户的个性化定制同时成为可能的原因。

赌注在于，当监督者有一个真实的计划和一套以业务为导向的特定指令（而不是每次运行时即兴发挥）时，编排和个性化会变得更容易控制。

**摩根大通在构建 Ask DAVID 过程中的两个关键教训：**

**准确率曲线**：首先，他们观察到了准确率随时间提升的一致模式。将通用模型应用于复杂的金融任务，初始准确率低于50%。提示工程和检索调优将性能推近80%，而工作流编排和子图则将其拉近90%。超过这个点，最后10%（"最后一公里"）需要人机协同。考虑到涉及数十亿美元资产规模，人工审查步骤是必不可少的。

**长评估周期**：第二，团队发现生成式AI项目通常比传统机器学习项目开发周期更短，但评估周期更长。为了提高可靠性，他们使用 LangSmith 独立评估每个子Agent以识别系统中的薄弱点，并选择与当前任务匹配的指标，例如摘要的简洁性或工具使用的轨迹评估。他们还将 LLM-as-judge 评估与人工审查相结合，使评估过程能够规模化而不压垮领域专家。

#### 桥水基金：PAT 与确定性研究Agent

桥水基金是一家系统化宏观对冲基金，PAT（袖珍分析师工具）是其应用 AI 团队构建的内部Agent，用于完成过去需要人工分析师数天甚至数周才能完成的深度分析工作。PAT 专为探索性研究而设计，让投资者能够追踪那些他们以前根本没有带宽去追问的问题。几个月前内部部署后，现在每天有数百名投资者在使用。

让这个项目与众不同的是它所建立的基础。桥水基金用了50年时间记录每笔交易背后的因果逻辑，并将方法论编纂成一个既机器可读又人类可读的专家系统。团队不必为了构建Agent而回过头去记录分析师如何工作——材料早已可用。PAT 直接利用公司的投资流程和框架，这是它能够模拟分析师实际工作的一个重要原因。

**逐用户的安全模型**

团队最初遇到了一个值得特别指出的安全问题，因为这个问题在基金行业之外也很常见。为了让 PAT 发挥作用，它需要访问其用户所能看到的一切。但是，不同的投资者有不同的访问级别。一个人可能能看到所有市场的完整持仓簿，而一个没有相应权限的分析师绝对不能看到。向错误的人泄露受保护的知识产权是不可接受的。

因此，与 Claude Code 这样的工具套件不同（每个人都使用相同的系统提示和相同的工具），桥水基金的每个人都获得一个独特的 PAT 版本，其范围完全由其身份被允许接触的上下文和工具所限定。

**有目的地拆分两个Agent**

架构被干净地分成两个Agent，因为投资者关心的是投资而非代码。聊天Agent完全专注于投资内容，底层的编码纯粹是一个在聊天体验中看不到的实现细节。

聊天Agent使用 LangGraph 实现，以获得持久状态、干净的取消和继续能力。它：

- 收集上下文并与用户达成对问题的共同理解
- 搜索网络和包含数百万非结构化文档的库（券商报告、财报电话会议记录、内部邮件，近实时更新）
- 然后搜索包含数千万个时间序列的数据库，这些序列是公司在五十年间建模积累的
- 搜索Agent使用普通的 RAG 排序，但关键的区别在于教会它们以人类研究员的方式检查数据：检查频率、时效性，以及数值是否与先验一致。仅这一项改变就将数据搜索准确率从大约50%提升到90%

**计划本身就是分析**

一旦问题明确，PAT 就开始规划，而团队的观点是：**计划本身就是分析**。一个高质量、详细的计划可以被一致地执行。计划命名了分析所需的每个数据框、每个的架构，以及最重要的是它们如何连接。这个规划步骤在墙上时间上是昂贵的，团队有意付出这个代价，因为它换来了执行阶段的价值。

**将Agent编码视为编译器问题**

团队坚信将Agent编码视为**编译器问题**而非Agent问题。编译器取用户代码并将其编译为机器代码；编码Agent取计划并将其编译为 Python。编译器几十年来一直在可靠地生成正确的、确定性的代码，团队借鉴了这些技术。

实践中：

- 计划被分解为任务，每个任务大致对应一个生成数据框的 Python 函数，附有足够的结构和语义细节，使得两个不同的 LLM 处理相同任务时产生语义等价、输出相同的代码
- 由于计划如此详细，代码生成可以跨任务并行运行，平均比顺序生成快约四倍。它还能超大规模化，所以一个20任务计划大约和一个3任务计划用时相当
- 生成的代码会根据其任务规范进行检查，如果不通过则重试，验证Agent在任务 DAG 上并行应用
- 正确性在架构层面得到保证，而不是委派给Agent的判断
- 结果是任何给定计划在两次独立运行中产生相同代码输出的概率为95%——实际上是一个确定性编码Agent

**工作如何复利**

一些细节对于如何自我增强至关重要。PAT 的分析输出会落回其输入来源的同一个时间序列数据库，这使得它们与公司几十年来人工上传的序列无法区分，也意味着任何 PAT 输出都可以成为后续分析的输入，所以人类和Agent可以在彼此的工作基础上持续建设。

执行后，PAT 会以你期望一个初级分析师检查工作的方式进行自我复核：验证数字看起来合理，图表看起来干净，如果有什么不对，它会在回馈之前诊断、优化并再次尝试。最终报告使用投资者使用的同一个内部图表库和文本页面。

**学习循环**

将一切串联起来的是 PAT 如何改进，这通过两种方式发生：

- **自主式**：Agent在后台运行，扫描已完成的对话以找出 PAT 出错的地方，从这些错误中开发经过人工审核的基准测试，并用它们来驱动对上下文的修改
- **显式**：认为 PAT 应该有不同行为的用户可以点击"教"按钮。一个Agent审查对话以查找行为错误、上下文缺口或用户引导，然后创建一个旨在失败的基准测试，在上下文仓库或工具套件上迭代直到通过，确认其余测试套件仍然通过，并以 Slack 消息的形式提交 Pull Request

无论哪种方式，下一个提出类似问题的用户都能直接获得改进后的版本。**改进是桥水基金生产工作流的一部分**。

#### Chime：Jade 以及他们如何让法律团队来写评估

Chime 是一家拥有 950 万会员的美国消费金融科技公司，Jade 是他们在 Deep Agents 上构建的始终在线的金融副驾驶。Jade 旨在帮助会员更聪明地消费、储蓄更多，并做出更好的财务决策。像任何面向消费者的Agent一样，它需要是有帮助的、安全的和可靠的。但因为它直接与用户就他们的财务进行交互，合规变得与用户体验本身同样重要。

这给团队带来了一个挑战。与合规团队协作的传统模式无法很好地映射到Agent开发上。通常，合规团队在启动阶段参与解释规则，然后在接近发布时再次出现以批准或阻止发布。如果问题在后期才浮出水面，团队可能会因重新审视早期决策而损失数周时间。在实践中，这也意味着团队可能在大部分开发周期中都在猜测自己是否在评估正确的风险。

Chime 的方法是将**评估作为工程和合规之间的对齐界面**。与其抽象地讨论宽泛的概念，两个团队都针对Agent行为的具体示例进行工作，并评估这些响应是否符合政策预期。

他们构建的流程遵循五个步骤：

1. **创建结构**：团队从领域分类法开始，如安全、安保、合规和正确性。在这些领域下是消费者保护或未经授权活动等类别，接着是特定风险，如未经授权的投资建议或未经授权的税务建议。

| 领域 | 安全 | 安保 | 合规 | 正确性 |
|------|------|------|------|--------|
| 类别 | | | 消费者保护和追索权 | 未经授权建议 |
| 风险 | | | 投资建议 | 法律建议 | 税务建议 |

2. **让法律团队定义风险**：法律和合规伙伴用他们自己的语言编写结构化的风险定义，包括被禁止的行为、法律背景、可接受的替代方案以及示例用户问题。

3. **引导数据集**：这些风险定义被输入 Giskard（一个开源红队框架），以生成对抗性测试问题。

4. **从相同的定义构建评估器**：结构化的风险定义被插入到模板化的 LLM-as-judge 提示中，为每个风险类别创建评估器。

5. **通过审查和反馈闭环**：法律审查员对评估结果进行标注，将他们的判断与评估器输出进行比较，并利用分歧来改进系统——无论是优化提示、调整数据集、调优评估器，还是澄清底层政策定义。

Chime 团队在 LangSmith 中运行这些工作流。结果可以根据受众在不同层级查看：工程师可以检查个别风险上的表现，合规团队可以监控更广泛的政策类别，领导层可以跟踪安全、安保、合规和正确性等领域的整体趋势。

当法律审查员与评估器结果不一致时，该反馈会直接反馈到开发循环中。团队可能会更新Agent提示、改进数据集、调整评估器，或甚至优化底层政策定义本身。

结合 LangSmith，这一流程帮助团队更早地发现合规问题，在整个开发过程中与法律伙伴更紧密地协作，并持续地而非等到发布关卡时才对系统建立信心。

**反馈飞轮**：风险定义 → 数据集生成器 → Agent响应 → 评估器 → 专家标注 → 反馈回到提示/数据集/评估器/政策

### 3. 构建者最关注的问题

在与金融服务领域的团队合作中，五个主题反复出现：

#### 1. 多Agent架构通过迭代逐步演进

单个 ReAct Agent 可以处理惊人数量的金融服务工作流，直到数据或任务的复杂性开始推至其极限。我们最常看到的模式与摩根大通的演进路径相似：团队从单个Agent开始，在不同数据类型需要不同处理方式时引入专业化，在多个Agent需要协调工作时添加监督者，最终将工作流拆分为子图。或者正如摩根大通正在做的，将监督者本身改造为带有真实计划和业务特定技能支撑的 Deep Agent。

最终的架构通常是多次迭代的产物，而非前期宏伟设计的成果，每次迭代都回应了上一版本暴露的局限性。

桥水基金的 PAT 从另一个角度展示了相同的情况。他们特意将聊天Agent和编码Agent分开，因为团队了解到投资者不想看到代码，而且当编码被当作编译器问题处理时，正确性更容易得到保证。

随着这些系统变得越来越复杂，团队需要一种方式来理解工作如何在Agent、工具、内存和检索层之间流动。LangSmith 通过在追踪链路中使这些关系可见来提供帮助，团队可以在一个平台上检查和评估这些工作流。

#### 2. 可观测性成为上线的先决条件

在金融服务领域，团队需要能够理解并解释系统是如何在事后得出结论的。这种期望不仅限于合规，还扩展到内部审计、客户支持和业务本身。正如一位在该领域构建的工程师所说：**你不能发布你无法解释的东西**。

桥水基金比大多数人走得更远。因为 PAT 的分析可能需要人工分析师花费数小时，所以分析必须是完全可诊断的——不仅对人类如此，对在后台运行、阅读追踪链路并验证每项计算是否正确的Agent也是如此。团队使用 LangSmith 实现追踪链路可见性，这使得人类和在后台运行的Agent都能逐步查看已完成的运行过程，精确了解它是如何得出结论的。

LangSmith 对整个学习循环至关重要，因为自主改进路径依赖于Agent能够阅读已完成的运行过程并找到出错的地方。

传统日志对于Agent系统来说是不够的。团队需要对提示、工具调用、检索到的上下文和模型输出的可见性，以便理解响应是如何生成的，并一致地调试问题。拥有一个统一界面来检查有问题的Agent行为、追踪实时系统性能，并对生产流量运行评估，可以帮助团队更早地发现回归问题，而不是在数周后通过支持升级才发现。

#### 3. 评估成为跨团队的共同语言

Chime 的方法可以广泛应用。在许多组织中，定义Agent可接受行为的人并非工程师。法律、风险、合规、品牌和运营团队都有系统需要遵循的要求。

评估为这些群体提供了一种具体的协作方式。与其讨论"公平披露"或"不安全建议"等抽象概念，团队可以针对示例进行工作：一个提示、一个响应，以及一个通过/失败的结果。拥有一个共享的工件使得技术和非技术团队之间的对齐变得容易得多。

许多团队还直接从生产追踪链路中构建评估数据集，使测试保持基于真实使用模式。Chime 在 LangSmith 中运行这些工作流，使用一个定义风险、生成对抗性示例并与法律伙伴一起审查评估器输出的流程作为持续评估循环的一部分。这是评估如何成为改进Agent系统的持续组成部分的一个示例，它根植于真实的生产行为，而不仅限于发布前的测试。

#### 4. 治理与合规向运行时迁移

这是与银行平台团队对话中最先出现的话题，它包含两个方面：治理和安全。离线评估很重要，但它只能捕获部署前的问题。团队同样关注实时运行期间会发生什么——无论是Agent整夜卡在重试循环中，敏感数据出现在日志中，还是在生产流量下工作流表现异常。因此，他们正在将治理推向更接近运行时本身。

具体而言，团队要求运行时控制，例如支出限制、人工审批检查点，以及在信息到达模型或被写入日志之前过滤或遮蔽敏感信息。

一个反复出现的需求是这些护栏是否能映射到 OWASP 的 LLM 和Agent应用指南；对于一些银行来说，护栏与 OWASP 风险类别之间的清晰映射是将项目从"有趣"推进到"可批准"的关键。

LangSmith 的 LLM 网关正是围绕这一层架构设计的，为团队提供一个集中的位置来应用运行时控制，同时在同一个运营界面上保持治理、可观测性和评估的连接。团队通常将这种整合视为重要，因为在追踪、评估、治理控制和监控分别位于独立系统中时，理解和调试Agent行为会变得更加困难。

两个相关的线索贯穿这些对话。首先，**开源模型**是银行平台团队的一个主要焦点，既因为其提供的控制力，也因为成本。其次，团队越来越多地**针对不同的护栏使用不同的模型**，而不是用同一个模型处理所有问题，因为模型在不同护栏任务上表现不均衡。一个我们经常听到的例子是 NeMo 在 PII 检测方面表现特别出色。一个允许团队将每个护栏路由到最擅长该任务的模型——而不是统一使用单一评判器——的网关，更符合这些团队实际想要的工作方式。

这里值得重新审视桥水基金的逐用户 PAT 版本，因为它显示了治理和安全在身份层面而非事后附加的方式得到处理。每个用户的Agent范围精确限定于他们被授权的数据和工具，这意味着访问控制不是Agent必须记住去应用的护栏——它是该用户正在与之对话的Agent本身的内置属性。

#### 5. 最后一公里的人机协同

摩根大通对"准确率曲线"的阐述与许多我们交谈的团队产生共鸣。早期版本的Agent可能表现不一致。检索调优和提示工程显著提高了可靠性。更结构化的流程和编排进一步推动了准确率。

但在高风险的金融工作流中，仍然存在需要人工审查的关键节点。与其试图完全移除人类，许多团队正在设计Agent处理重复性工作、人类在判断或问责至关重要的时刻保持参与的系统。

这一要求改变了运行时架构本身。团队需要能够暂停工作流、将输出路由到审查、收集标注、然后干净地恢复执行的系统。LangSmith Deployment 正是围绕这些长时间运行和人机协同的工作流设计的，为审批、协调和Agent执行提供持久的运行时环境。在实践中，**人工审查正在成为生产工作流的一部分**，而不是一个事后叠加的独立运营流程。

---

### 4. 总结

当今将Agent投入生产环境的金融服务团队，并不将部署视为过程的终点。他们正在构建覆盖整个**Agent开发生命周期**的持续工作流，包括构建、测试、部署、监控和治理。

这在我们讨论过的团队中呈现出不同形式。摩根大通将 Ask DAVID 从一个简单的 ReAct Agent 演进为多Agent系统，现在正在将其监督者改造为由专有私人银行技能驱动的 Deep Agent。桥水基金基于 50 年积累的方法论构建了 PAT，将其拆分为聊天Agent和确定性编码Agent，并建立了一个将真实交互转化为基准测试的学习循环。Chime 将评估打造成了工程、法律和合规团队之间的共享运营流程。

在这三个案例中，底层模式是相似的。团队正在构建可以被持续评估、在生产中监控、并在新行为出现时持续优化的系统。可观测性、评估、运行时治理和人工审查不再是开发后期才加入的独立步骤。它们正在成为在受监管环境中发布Agent的运营基础。

这些系统没有一个已经完成。架构继续演进，评估循环继续改进，治理模型继续成熟。这些团队已经构建了允许他们安全地、增量式地改进Agent的工作流，并随着时间推移获得清晰的运营反馈。**这在金融服务领域，越来越成为生产级Agent开发的样貌**。
