整体方向我认为是**对的，而且定位已经比 "Enterprise Search + RAG" 高一个层次**。

但是如果目标是一个**可以真正落地的 Agent Skill**，我会做几个调整。它们不是增加功能，而是**收敛概念、减少重复、强化核心能力**。避免给人一种"什么都想做"的感觉。

---

# 1. 最大的问题：Research Pipeline 有重复

目前你有两个 Pipeline：

第一个：

```text
Research Goal
→ Hypothesis Planning
→ Evidence Collection
→ Evidence Correlation
→ Gap Detection
→ External Verification
→ Report
```

后面又来了：

```text
Planner
→ Question Decomposition
→ Evidence Collector
→ Entity Resolver
→ Cross Validation
→ Gap Detector
→ Hypothesis Generator
→ Report Writer
```

实际上这是同一件事。

建议保留一个即可。

例如：

```mermaid
flowchart LR

Goal
--> Planner
--> Evidence_Collection
--> Entity_Resolution
--> Evidence_Correlation
--> Gap_Analysis
--> External_Verification
--> Report
```

更加清晰。

Entity Resolution 放在 Collection 后面。

Correlation 放在 Resolution 后面。

这符合 Investigation 的实际流程。

---

# 2. 不建议一直强调 Multi-Agent

这是我觉得最容易 over engineering 的地方。

实际上：

Research Agent

下面只是多个 Specialist。

完全可以只是：

```
Planner

Collector

Resolver

Analyzer

Reporter
```

至于内部是不是多个 Agent，

还是 Workflow，

还是 DAG，

还是 LangGraph，

都不重要。

Proposal 不需要强调。

否则 Review 的人第一反应：

> 为什么要 Multi-Agent？

然后开始 endless discussion。

最好一句：

> The workflow can be implemented as a single orchestrated agent or a collection of specialized agents, depending on deployment requirements.

一句带过。

# 3. Entity Resolution 应该升级成 Identity Resolution

Entity Resolution 偏数据库。

Identity Resolution 更符合 Enterprise。

例如：

```
RiskConcile

riskconcile-api

Vendor 28391

RC

RiskConcile Ltd.
```

最后形成：

```
Canonical Identity
```

以后所有 Connector 都返回：

```
Identity
```

这比 Entity Resolution 更符合 Gartner、MDM、Data Catalog 的术语。

# 4. Correlation 才是真正核心

全文其实一直在讲 Correlation。

建议单独拉出来。

例如：

> Enterprise Search answers:

```
Where is the document?
```

Research Agent answers：

```
How are these pieces of evidence related?
```

例如：

```
Vendor

↓

Application

↓

Repository

↓

Team

↓

Incident

↓

Risk

↓

Control
```

Correlation 才是整个 Proposal 最大的 Differentiator。

现在它写得还不够突出。

# 6. Research Graph 不要放最后

Research Graph 是全文最重要的创新点。

建议提前。

整个流程应该变成：

```
Research Question

↓

Identity Resolution

↓

Research Graph

↓

Evidence Correlation

↓

Gap Detection

↓

Verification

↓

Report
```

因为：

Graph 不是最终结果。

Graph 是整个 Investigation 的 Working Memory。

这点非常重要。

# 7. Report 建议增加 Evidence Traceability

企业最怕：

> AI 怎么知道的？

建议 Report 增加：

```
Finding

Evidence

Confidence

Conflicts

Unknown

Recommendation
```

例如：

| Section          | 内容                      |
| ---------------- | ----------------------- |
| Finding          | RiskConcile 已在 12 个系统使用 |
| Evidence         | Jira、LeanIX、GitHub      |
| Confidence       | High                    |
| Missing Evidence | Contract 未找到            |
| Recommendation   | 联系 Vendor Manager       |

这样企业可信度会高很多。

# 10. 最后建议增加一张 "Why not RAG?" 的对比

这是 Proposal 最缺的一张图。

例如：

| Traditional RAG    | Enterprise Research Agent                |
| ------------------ | ---------------------------------------- |
| Retrieve documents | Investigate a business question          |
| Return passages    | Produce evidence-backed findings         |
| Document-centric   | Entity- and evidence-centric             |
| Single source      | Multi-source correlation                 |
| Q&A                | Research workflow                        |
| Limited reasoning  | Hypothesis, validation, and gap analysis |

这一页几乎可以决定 Reviewer 是否立刻理解你的定位。

---

## 我认为还可以增加一个核心概念：Evidence-Driven Research

如果全文只能保留一个关键词，我甚至不会选 **RAG**、**Research Graph** 或 **Multi-Agent**，而会选 **Evidence**。

整个 Agent 的价值其实可以浓缩为一句话：

> The Enterprise Research Agent is an evidence-driven investigation system. Rather than retrieving documents, it identifies canonical identities, correlates evidence across enterprise systems, validates findings against external knowledge, and produces traceable research reports with explicit confidence, conflicts, and knowledge gaps.

这句话几乎概括了全文的核心思想，而且没有引入额外复杂度，也不会给人过度设计的印象。它还能自然地串联起 Identity Resolution、Research Graph、Evidence Correlation 和 Research Report，使整个 Proposal 的主线更加聚焦。
