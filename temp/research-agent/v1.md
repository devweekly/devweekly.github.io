我觉得这个 Agent Skill，如果只是做 **Enterprise Search + RAG**，价值其实一般。

真正值得做的是一个 **Enterprise Research Agent（企业研究员）**，它不是回答问题，而是**完成一项研究任务（Research Task）**。

金融服务公司其实每天都有大量这种需求：

> "研究一下 RiskConcile，看看我们是否已经合作过。"
>
> "ESG 到底影响哪些业务系统？"
>
> "Open Banking 在公司里面有哪些项目？"
>
> "MAS TRM 新规需要影响哪些 Application？"

这些都不是搜索，而是 Investigation（调查）。

# 我建议定位成 Enterprise Research Agent

不要叫 Search。叫 Enterprise Research Agent 或者 Domain Research Agent。因为它最终输出的是 Research Report 而不是 Search Result。

整个 Agent 可以定义成：

```mermaid
flowchart TD
    Input --> Research Goal
    Research Goal --> Hypothesis Planning
    Hypothesis Planning --> Evidence Collection
    Evidence Collection --> Evidence Correlation
    Evidence Correlation --> Gap Detection
    Gap Detection --> External Verification
    External Verification --> Research Report
```

这和 Deep Research 非常类似，但是数据源主要来自企业内部。

# 我建议的数据源

实际上可以分成四层。

| Layer                   | Source                      | 作用     |
| ----------------------- | --------------------------- | ------ |
| Knowledge               | Confluence、SharePoint、Wiki  | 文档     |
| Engineering             | GitHub、GitLab、Azure DevOps  | 代码     |
| Operations              | Jira、ServiceNow             | 实际运行情况 |
| Enterprise Architecture | LeanIX、DataHub、OpenMetadata | 企业模型   |

第五层 External 包括：

* Vendor 官网
* Gartner
* Microsoft
* AWS
* GitHub
* arXiv
* SEC Filing
* 新闻
* Whitepaper

例如研究 RiskConcile 时，不仅要看官网，还应综合公开资料了解其定位、产品能力以及行业评价。([RiskConcile Group][1])

# 重点不是搜索，而是 Correlation

真正困难的是：不同系统的数据没有关联。

例如：

| 系统 | 标识 |
|------|------|
| Confluence | `RiskConcile` |
| GitHub | `riskconcile-api` |
| LeanIX | `Vendor=Riskconcile` |
| ServiceNow | `Vendor ID = 28391` |
| Jira | `RC Migration` |

它们其实都是同一个东西。所以 Agent 第一件事情不是搜索。而是 Entity Resolution。

建立：

- RiskConcile
- Alias
- Riskconcile
- RC
- riskconcile-api
- Vendor 28391

形成一个 Entity。以后所有搜索都围绕 Entity 展开。这一层甚至比 RAG 更重要。

# 我建议增加 Enterprise Semantic Layer

很多人把 Semantic Layer 理解成 BI 的指标层。

但对于 Agent，我建议定义成 Research Entity 类型：

- People
- Application
- Vendor
- Capability
- Project
- Business Process
- Technology
- Risk
- Policy
- Regulation
- Incident
- Ticket
- Repository
- Document

所有 Connector 返回的数据最终都映射成统一 Entity。

例如：

| 来源 | 原始对象 | 映射 Entity |
|------|---------|------------|
| GitHub | Repo | Repository Entity |
| Jira | Epic | Project Entity |
| Confluence | Page | Document Entity |
| LeanIX | Application | Application Entity |

这样 Agent 不关心数据来源。它只关心 Entity。

# Skill 不应该一个 Connector 一个 Skill

很多人会这样设计：Github Skill、Jira Skill、Confluence Skill、LeanIX Skill。

我不建议。应该是：

```mermaid
flowchart TD
    Research Skill --> Github Connector
    Github Connector --> Jira Connector
    Jira Connector --> LeanIX Connector
    LeanIX Connector --> Confluence Connector
```

Connector 是 Infrastructure。Skill 是 Research。

# ESG 的流程又不同

ESG 属于 Topic Research。例如：

Agent 会自动去：

- Jira 找 ESG Epic
- ServiceNow 找 ESG Service

然后去外部：ECB、ESMA、ISSB、GRI、SASB、新闻、GitHub、Paper，全部融合。

# 我觉得最值得做的是 Research Graph

不要直接做 Vector。

建议建立一个轻量级的研究图谱（Research Graph）：

Entity 属性：

- id
- type
- aliases
- summary

Relationship：

```mermaid
graph LR
    Vendor -->|uses| Application
    Application -->|implemented_by| Repository
    Repository -->|owned_by| Team
    Team -->|belongs_to| Capability
    Capability -->|supports| Business Process
```

注意，它**不是 Ontology**，也不是完整的 Knowledge Graph，而是围绕"研究任务"构建的实体关系层。你的 Agent 在每次研究时，都可以不断补充和修正这个 Graph。

## 我认为这个项目真正的价值

如果让我给这个项目定位，我不会定位为 Enterprise Search，也不会定位为 Enterprise RAG，而会定位为 **Enterprise Research Intelligence Platform**。

它的目标不是回答"某个文档里写了什么"，而是自动完成过去需要高级架构师、Enterprise Architect、Risk Analyst 或 Business Analyst 花几小时甚至几天完成的调查研究工作。它能够跨 Confluence、GitHub、Jira、LeanIX、ServiceNow 等内部系统收集证据，结合外部公开信息进行交叉验证，自动识别冲突与缺口，并输出带有引用依据、影响分析和建议的研究报告。这类能力比传统 RAG 更接近 OpenAI Deep Research 的企业版，但数据源和推理重点聚焦于金融服务企业的内部知识与架构资产。
