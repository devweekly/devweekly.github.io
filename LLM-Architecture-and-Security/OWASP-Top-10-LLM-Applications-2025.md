# OWASP Top 10 for Large Language Model Applications (2025)

> **来源**: [OWASP GenAI Security Project](https://genai.owasp.org/) | [GitHub Repository](https://github.com/OWASP/www-project-top-10-for-large-language-model-applications)
>
> OWASP Top 10 for LLM Applications 是一个面向开发者和应用安全从业者的标准意识文档，代表了关于 LLM 应用最关键安全风险的广泛共识。

---

## LLM01:2025 提示注入 (Prompt Injection)

提示注入漏洞发生在用户提示以非预期方式改变 LLM 行为或输出时。这些输入即使人类无法感知，也可能影响模型——提示注入不需要人类可见/可读，只要内容被模型解析即可。

### 直接提示注入
用户的提示输入直接以非预期方式改变模型行为。

### 间接提示注入
LLM 接受来自外部来源（如网站或文件）的输入。外部内容中的数据被模型解释时，可能导致非预期行为。

**缓解措施**：
- 在系统提示和输入处理中构建防护措施
- 持续更新模型训练和安全机制来防范越狱攻击

---

## LLM02:2025 敏感信息泄露 (Sensitive Information Disclosure)

敏感信息可能影响 LLM 及其应用上下文，包括个人身份信息(PII)、财务详情、健康记录、商业机密、安全凭证和法律文件。

LLM 嵌入应用时，可能通过输出暴露敏感数据、专有算法或机密细节，导致未经授权的数据访问、隐私侵犯和知识产权泄露。

**常见风险示例**：
1. PII 泄露
2. 专有算法暴露
3. 敏感商业数据泄露
4. 安全凭证泄露

**缓解措施**：
- 执行充分的数据脱敏
- 提供清晰的 Terms of Use 政策
- 通过系统提示添加限制

---

## LLM03:2025 供应链 (Supply Chain)

LLM 供应链易受多种漏洞影响，可能损害训练数据、模型和部署平台的完整性。除了传统软件漏洞（代码缺陷和依赖问题），ML 中的风险还扩展到第三方预训练模型和数据。

LoRA、PEFT 等微调方法以及 Hugging Face 等平台引入了新的供应链风险。设备端 LLM 的出现也增加了攻击面。

**常见风险**：
1. 传统第三方包漏洞（过时或弃用组件）
2. 模型许可风险
3. 数据集污染
4. 预训练模型后门

**缓解措施**：
- 审查第三方模型和数据源
- 使用 OWASP CycloneDX 或 ML-BOM 跟踪组件
- 定期审计更新依赖

---

## LLM04:2025 数据和模型投毒 (Data and Model Poisoning)

数据投毒发生在预训练、微调或嵌入数据被操控以引入漏洞、后门或偏见时。可能针对 LLM 生命周期的不同阶段（预训练、微调、嵌入）。

**常见风险**：
- 攻击者通过有害数据导致偏见输出（Split-View Poisoning、Frontrunning Poisoning）
- 恶意 pickling 嵌入恶意代码
- 后门触发机制（sleeper agent）

**缓解措施**：
1. 使用 OWASP CycloneDX 或 ML-BOM 跟踪数据来源
2. 严格审查数据供应商
3. 实施严格的沙箱隔离
4. 使用数据版本控制(DVC)
5. 红队测试和对抗训练
6. 监控训练损失，检测异常输出
7. 推理阶段整合 RAG 和 grounding 技术

---

## LLM05:2025 不当输出处理 (Improper Output Handling)

指对 LLM 生成输出在传递给下游组件之前未能进行充分验证、净化和处理。成功利用可导致 XSS、CSRF、SSRF、权限提升或远程代码执行。

**常见风险**：
1. LLM 输出直接进入系统 shell（exec/eval），导致 RCE
2. LLM 生成的 JavaScript/Markdown 被浏览器解释，导致 XSS
3. LLM 生成的 SQL 查询无参数化，导致 SQL 注入
4. LLM 输出用于构造文件路径且未净化，导致路径穿越
5. LLM 生成内容用于邮件模板未转义，导致钓鱼攻击

**缓解措施**：
- 零信任模型输出，视模型如同普通用户
- 遵循 OWASP ASVS 进行输入验证和输出编码
- 上下文感知的输出编码
- 使用参数化查询
- 严格的 CSP 策略
- 健全的日志和监控

## LLM06:2025 过度代理 (Excessive Agency)

LLM 系统通常被授予一定程度的代理能力——调用函数或通过扩展与其他系统交互。过度代理使得在响应意外、模糊或被操控输出时能够执行破坏性操作。

**根本原因**：
- 功能过多
- 权限过大
- 自主性过高

**常见风险**：
- LLM Agent 拥有超出系统预期的功能权限
- 扩展在开发阶段试用后未被限制
- 过于开放的权限设置

**缓解措施**：
- 最小权限原则
- 限制 Agent 可调用的功能和权限
- 减少不必要的自主性

---

## LLM07:2025 系统提示泄露 (System Prompt Leakage)

系统提示用于引导模型行为，但可能包含不应被发现的敏感信息。**系统提示不应被视为秘密，也不应用作安全控制**。凭证、连接字符串等不应包含在系统提示中。

**核心要点**：系统提示本身的泄露不构成真正风险——真正的安全风险在于底层元素：敏感信息泄露、系统护栏绕过、权限分离不当等。

**常见风险**：
- 系统提示暴露凭证、API 密钥、数据库名称
- 暴露内部决策流程
- 暴露角色结构和权限级别

**缓解措施**：
- 不在系统提示中嵌入任何敏感信息（API 密钥、认证密钥、数据库名、用户角色等）
- 将行为控制外部化到 LLM 之外的系统
- 权限分离等关键控制不能在 LLM 中执行
- 对不同访问级别的任务使用多个 Agent

## LLM08:2025 向量和嵌入弱点 (Vector and Embedding Weaknesses)

在利用 RAG 的系统中，向量和嵌入的生成、存储或检索方式存在安全风险。可能被利用来注入有害内容、操控模型输出或访问敏感信息。

**常见风险**：
- 访问控制不足导致嵌入中的敏感信息泄露
- 多租户环境中的上下文泄露
- 嵌入反转攻击恢复源信息
- 数据投毒
- RAG 改变基础模型行为（如共情能力下降）

**缓解措施**：
- 细粒度访问控制和权限感知的向量存储
- 知识源的数据验证管道
- 数据分类和标记
- 详细的不可变检索活动日志

---

## LLM09:2025 错误信息 (Misinformation)

LLM 产生看似可信但虚假或误导性信息。主要原因包括幻觉（模型用统计模式填补训练数据空白）和过度依赖（用户过度信任 LLM 输出而不验证）。

**常见风险**：
- 模型生成错误陈述，导致用户基于虚假信息做决策（如 Air Canada 聊天机器人案例）
- 在医疗/法律等敏感领域生成无根据断言（如 ChatGPT 编造虚假法律案例）
- 建议使用不安全或不存在的代码库（包幻觉攻击）

**缓解措施**：
- 使用 RAG 增强输出可靠性
- 微调和嵌入提升输出质量
- 鼓励用户交叉验证
- 自动验证关键输出
- 明确向用户传达风险和局限性
- 安全编码实践

---

## LLM10:2025 无限制消费 (Unbounded Consumption)

当 LLM 应用允许用户进行过度和不受控制的推理时，导致拒绝服务(DoS)、经济损失、模型盗窃和服务降级。云环境中 LLM 的高计算需求使其容易受到资源利用和未经授权使用的影响。

**常见风险**：
1. **变长输入洪流**：大量不同长度输入耗尽资源
2. **钱包拒绝服务 (DoW)**：利用按使用付费模式，造成不可持续的经济负担
3. **连续输入溢出**：持续发送超过上下文窗口的输入
4. **资源密集型查询**：触发最昂贵的计算流程
5. **模型提取**：通过 API 精心构造输入收集输出以复制模型
6. **功能性模型复制**：使用目标模型生成合成训练数据微调另一个模型
7. **侧信道攻击**：利用输入过滤绕过获取模型权重和架构信息

**缓解措施**：
- 输入验证（限制输入大小）
- 限制 logits/logprobs 暴露
- 速率限制和用户配额
- 动态资源分配管理
- 超时和节流
- 沙箱技术
- 全面日志、监控和异常检测
- 水印技术
- 优雅降级
- 对抗性鲁棒训练
- RBAC 和最小权限原则
- 集中式 ML 模型清单
